Dump Servo — Positional Mode

The robot's dump mechanism uses a positional servo connected to Control Hub Servo Port 0. When you press X, the servo flips between its home position and its dump position. This lesson covers what that servo is, how to wire and configure it, and exactly how the code works.

Section 1 — What Is a Servo? (Quick Review)

A servo is a small motor with a built-in position sensor and control circuit. Unlike a plain DC motor that just spins, a servo actively moves to a specific angle and holds it there. Inside are three parts working together: a DC motor, a gear train, and a potentiometer that tracks the shaft position.

If this is your first time with servos, read the Indexer Servo lesson first — it goes into detail on how PWM signals, potentiometers, and gear ratios work.

💡 Key Idea
A servo moves to a position and holds it. A DC motor just spins. That built-in holding ability is what makes servos useful for arms, dumps, and claws.

Section 2 — Positional Mode vs CR Mode

The goBILDA 2000 Series servo is a dual-mode servo — it can be either positional or continuous rotation (CR). The mode is set with a small programmer tool before you install it on the robot. Our dump servo is in positional mode.

ModeWhat you controlPotentiometer used?Typical use
Positional (Normal) Target angle (0° – 300°) Yes — holds position Dump arms, claws, gates, wrist joints
Continuous Rotation (CR) Speed and direction No — ignored Intake rollers, indexers, feed wheels

In positional mode, you tell the servo where to go and it figures out how to get there. In CR mode, you tell it how fast to spin and it never stops unless you command it to.

💡 Key Idea
setPosition() is for positional servos. setPower() is for CR servos. Using the wrong one compiles fine but does not do what you expect.

Section 3 — goBILDA Servo Models: Torque vs Speed

goBILDA makes two main variants of the 2000 Series servo. The difference is in the gear train inside — more gears means more torque but slower speed.

Torque (25-2)Speed (2-1)
Stall Torque (6V)300 oz-in (21.6 kg·cm)~40 oz-in

The dump arm is loaded — it holds weight against gravity at its extended position. That is why we use the Torque variant. If this were a simple ball gate that just opens and closes with no real load, the Speed variant would be faster and fine.

💡 Key Idea
Pick the Torque servo when your mechanism holds something heavy or works against gravity. Pick the Speed servo when you need fast movement and the load is light.

Section 4 — Connecting the Servo

Positional servos plug directly into the servo ports on the Control Hub or Expansion Hub. Each port has three pins: ground, power, and signal.

[Control Hub diagram — see REV documentation]

  • Servo ports are labeled 0 – 5 on the Control Hub
  • The dump servo is on Servo Port 0
  • The 3-pin servo connector only fits one way — the brown wire (ground) goes to the side marked with a minus sign
  • Each servo port is powered from the hub's 5V rail — no separate power wiring needed
⚠️ Watch Out
Do not plug a servo into a motor port — the voltage is 12V and will immediately destroy the servo. Servo ports are the smaller connectors on the top row of the Control Hub.

Section 5 — Driver Hub Configuration

The Driver Hub needs to know the servo exists before your code can use it. You add it in the robot configuration file.

  1. On the Driver Hub, tap the three-line menu and open Configure Robot
  2. Select your existing configuration (or create one if this is the first time)
  3. Under the Control Hub, tap Servo Controller
  4. Find Port 0 and tap it
  5. Set the type to Servo (not CRServo)
  6. Set the name to dump_servo — this must exactly match the string in your code
  7. Tap Done and save the configuration
💡 Key Idea
The name in the Driver Hub config and the name in hardwareMap.get() must be identical — same spelling, same capitalization. A mismatch causes a runtime crash when the OpMode starts.

Section 6 — The Code

The servo code in MecanumDrive.java has four parts: an import, variable declarations, initialization, and the loop. We will walk through each one.

6.1 — The Import

Every hardware type needs an import at the top of the file so Java knows where to find its class.

📄 From MecanumDrive.java
import com.qualcomm.robotcore.hardware.Servo;
This line pulls in the Servo class from the FTC SDK. Without it, Java sees the word Servo and has no idea what it means — you will get a "cannot find symbol" compile error.

6.2 — Variables and Initialization

These lines appear before waitForStart(), in the setup area of the OpMode.

📄 From MecanumDrive.java — Servo Setup
// Track previous X button state for rising edge detection
boolean prevX = false;

// Track current servo state
boolean servoOut = false;

// Get the servo from the hardware map
Servo dumpServo = hardwareMap.get(Servo.class, "dump_servo");

// Set the initial servo position
dumpServo.setPosition(0.0);

Four things are happening here:

  • prevX — remembers whether X was pressed on the previous loop iteration (explained in section 6.3)
  • servoOut — remembers whether the servo is currently out or home (explained in section 6.4)
  • hardwareMap.get(Servo.class, "dump_servo") — looks up the servo by name from the Driver Hub configuration
  • setPosition(0.0) — commands the servo to its home position immediately when the OpMode starts, so it does not start in some unknown position

6.3 — Rising Edge Detection

This is the most important concept in this lesson and it applies to any button-triggered action in FTC.

The while (opModeIsActive()) loop runs roughly 50 times per second. If you check gamepad1.x directly and act on it, pressing X for just half a second means the condition is true for about 25 consecutive loops. The servo would toggle 25 times — too fast to see, ending up back where it started.

Rising edge detection solves this. Instead of asking "is X pressed right now?", you ask "did X just become pressed?" — meaning it was not pressed last loop but is pressed this loop. That transition (false → true) is called the rising edge, and it only happens once per button press, no matter how long you hold it.

Loopgamepad1.xprevXx && !prevXAction

The code for the check is:

📄 The Rising Edge Check
if (gamepad1.x && !prevX) {
    // only runs once, on the first loop where X is pressed
}
prevX = gamepad1.x;  // save this frame for next loop
prevX = gamepad1.x runs every loop — not just inside the if-block. If you put it inside the if-block, prevX only updates when the button fires, which breaks the detection.
💡 Key Idea
Rising edge detection fires an action once when a button is first pressed, regardless of how long it is held. The check is: current && !previous.

6.4 — The State Machine

The dump servo has two positions: Home (dumper is retracted) and Out (dumper is extended). Pressing X switches between them. That is a state machine.

A state machine is a program that:

  1. Tracks which state it is currently in
  2. Transitions to a different state when a specific input occurs

Ours has two states and one input: 

State: HOME  →  press X  →  State: OUT
State: OUT   →  press X  →  State: HOME

The state is stored in a boolean called servoOut. When we enter the rising-edge block, we flip it with servoOut = !servoOut, then immediately command the servo to the matching position:

📄 From MecanumDrive.java — Dump Servo Code
if (gamepad1.x && !prevX) {
    servoOut = !servoOut;                                  // flip the state
    dumpServo.setPosition(servoOut ? 4.0 : 0.0);          // move to matching position
}
prevX = gamepad1.x;

telemetry.addData("dumpServo", servoOut ? "Out" : "Home"); // show state on Driver Hub

The ? is a ternary operator — a compact if/else: condition ? value_if_true : value_if_false. So servoOut ? 4.0 : 0.0 means "if servoOut is true, use 4.0; otherwise use 0.0."

💡 Key Idea
The state machine stores which position the servo should be in so it survives across loops. Without a state variable, you have no way to know whether to move to Home or Out — you would need to physically read the servo's position, which only tells you where it ended up last time.

6.5 — Position Mapping: 0.0 and 4.0

setPosition() takes a value between 0.0 and 1.0. That range maps to the servo's full physical travel — for a goBILDA Torque servo in default mode, that is 0° to 300°.

Position → Angle (goBILDA Torque, 300° range):

So what about setPosition(4.0) in the code?

The FTC SDK clamps any value outside 0.0–1.0. A value of 4.0 gets treated the same as 1.0 — the servo moves to its maximum position (300°). The programmer used 4.0 as a way of saying "go as far as possible in the out direction." The result is identical to writing setPosition(1.0).

💡 Key Idea
  • setPosition(0.0) = 0° (home)
  • setPosition(1.0) = 300° (full out)
  • setPosition(4.0) = clamped to 1.0 = 300° (same result)
  • To hit a specific angle, divide it by 300: 120° → setPosition(0.4)
💜 Hint
The dump servo probably does not need to go a full 300°. Once you have the code working, tune the 4.0 (or equivalent) down toward 1.0, then toward lower values until the dump position is exactly where you need it mechanically. Same for the home position — 0.0 might push the servo hard against a mechanical stop, so you may want to raise it slightly.

2. Click Recompile on the Driver Hub.

4. Click INIT, then START.
✅ To Do
Test the servo:
1. Press X once — servo should move to the out position. Telemetry shows dumpServo: Out.
2. Press X again — servo returns to home. Telemetry shows dumpServo: Home.
3. Hold X for a full second, then release. It should have toggled exactly once, not bounced back and forth.
4. Drive around and press X at the same time — the servo should toggle regardless of what the robot is doing.
💜 Hint
If the servo toggles twice per press, check that prevX = gamepad1.x is outside the if-block, not inside it. If it is inside the block it only updates when the button fires, which breaks the edge detection.
