IMU Heading

What is an IMU?

Your robot has a tiny sensor called an IMU — an Inertial Measurement Unit. It contains a gyroscope that measures how fast and how far the robot has rotated. By tracking rotation over time, it can tell you the robot's heading — the direction it is currently facing.

💡 Key Idea
An **IMU** (Inertial Measurement Unit) uses a gyroscope to track

how far your robot has rotated. Its main output is heading — the angle the robot is facing.

Think of heading like a compass. When the robot starts, its heading is 0°. Turn it counter-clockwise (to the left) and the heading increases toward +180°. Turn it clockwise (to the right) and the heading decreases toward -180°.

💡 Key Idea
The FTC IMU reports heading from **0° to +180°** (counter-clockwise)

and 0° to -180° (clockwise). It does not go from 0 to 360.

Heading is measured in either degrees or radians. Both describe the same thing — angles around a circle — just with different numbers:

  • +90° = π/2 radians ≈ 1.571 (quarter turn left)
  • +180° = π radians ≈ 3.14159 (half turn)
  • -90° = -π/2 radians ≈ -1.571 (quarter turn right)

Most FTC code uses radians because the math library functions (Math.cos, Math.sin) expect radians.

Why Does Heading Matter?

Without an IMU, your robot drives relative to itself. Push the stick forward and it goes wherever the front is pointing. That works fine until the robot gets bumped, or the driver loses track of which way is "forward."

With an IMU, you can do field-centric driving. The joystick directions stay fixed relative to the field — push forward and the robot goes away from you, no matter which way it is facing. The competition code already uses this technique.

You can also use heading in autonomous mode to drive straight lines, make precise turns, and correct for drift.

💡 Key Idea
**Field-centric driving** uses the IMU heading to keep joystick

directions fixed relative to the field, not the robot.

The Turntable and Compass Rose

Look at the shiny metal disc on the test stand — that is a turntable. The robot sits on top of it.

Around the turntable you will see a compass rose — degree and radian markings printed on the table surface. Notice the labels: 0°, 90°, 180°, and -90°. There is a small red chevron pointing at the 0° position, and a red ring around the turntable marking its edge.

You will also notice a green line extending from the center of the turntable. This is the heading indicator — it points in the direction the robot is currently facing. Floating above its tip is a glowing green number — that is the live IMU heading readout in degrees, just like what your code will see.

On the right side of the table you will see a dark control box with two buttons labeled ROTATE:

  • + (left button) — rotates counter-clockwise (positive heading)
  • – (right button) — rotates clockwise (negative heading)
💡 Key Idea
Counter-clockwise rotation gives **positive** heading values.

Clockwise rotation gives negative heading values. The green floating number shows the current IMU heading in real time.

Talking to the IMU in Code

An API (Application Programming Interface) is a set of ready-made commands that let your code talk to a piece of hardware without worrying about the details inside. Think of it like a TV remote: you press "volume up" without knowing how the electronics work. The FTC SDK gives you an API for the IMU so you can read the heading with just a few lines.

You access the IMU through the hardware map, just like motors and servos. Here is the basic pattern: 

// Get the IMU from the hardware map


IMU imu = hardwareMap.get(IMU.class, "imu");




// Initialize it with default parameters
imu.initialize(new IMU.Parameters());

Once initialized, you read the current heading like this: 

double heading = imu.getRobotYawPitchRollAngles()


                    .getYaw(AngleUnit.RADIANS);

That looks like a lot of chained method calls, but it breaks down simply:

  1. getRobotYawPitchRollAngles() — returns an object with

yaw, pitch, and roll

  1. .getYaw(AngleUnit.RADIANS) — extracts just the yaw

(heading) in radians

You can also reset the heading back to zero at any time: 

imu.resetYaw();

This is useful when the robot drifts over time or you want to re-zero the heading at the start of a match. We will explore what reset does shortly.

  • AngleUnit.RADIANS — tells the IMU to return values in radians

Part 2 — Initialize the IMU

Now add the IMU initialization code. This goes before waitForStart(), alongside the motor setup.

✅ To Do
Click the link below to jump to the right spot:
[ Button Actions : Variables ]
✅ To Do
Right below the button action variables section, add the

IMU setup code from the green block:

>> Type This Code
// ===============================================


// IMU Heading
// Do not remove this comment
// Add your IMU setup code below
// ===============================================
IMU imu = hardwareMap.get(IMU.class, "imu");
imu.initialize(new IMU.Parameters(
    new RevHubOrientationOnRobot(
        RevHubOrientationOnRobot.LogoFacingDirection.UP,
        RevHubOrientationOnRobot.UsbFacingDirection.FORWARD
    )
));

This gets the IMU from the hardware map and initializes it. RevHubOrientationOnRobot tells the IMU how the Control Hub is mounted — logo facing UP and USB ports facing FORWARD. This matches our simulator robot, and it is the most common mounting on real robots too. If your real robot has the Hub mounted differently, you would change these two directions to match.

Part 6 — Understanding IMU Reset

Now let's see what resetYaw() actually does. This is an important concept to understand.

✅ To Do
1. Rotate the robot about **90° clockwise** so the heading reads

roughly -90°. Look at the compass rose — the green heading indicator points to the right.

  1. Now press the right bumper on the gamepad.

Watch what happens:

  • The green heading number snaps back to 0°
  • The compass rose rotates — its 0° mark and red chevron

shift to match the robot's new "forward" direction

  • The green line stays pointing the same direction — the

robot didn't physically move

💡 Key Idea
`resetYaw()` does **not** move the robot. It tells the IMU

"wherever I am pointing right now is the new 0°." The compass rose shifts to show this — its 0° mark moves to the robot's current direction.

✅ To Do
Now spin the turntable again. Notice that the heading counts

from the new zero point, not the original one. The compass rose markings show the updated reference frame.

This is how the competition code uses resetYaw() — at the start of a match, the driver presses the right bumper so that the robot's current facing direction becomes 0°, no matter how it was placed on the field.

  • Heading is the angle the robot is facing (yaw)
  • The IMU reports 0° to +180° counter-clockwise and

0° to -180° clockwise — not 0° to 360°

  • FTC code typically uses radians (the math functions

expect them)

  • hardwareMap.get(IMU.class, "imu") gets the sensor
  • imu.getRobotYawPitchRollAngles().getYaw() reads the heading
  • imu.resetYaw() makes the current direction the new 0°
  • The green heading indicator and floating green number
