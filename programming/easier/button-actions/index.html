Button Actions: Intake

What Is a Robot Action?

Driving is only half the job. During a match your robot also needs to interact with the field — picking up game elements, scoring them, and releasing them. Each of these tasks is an action.

💡 Key Idea
An **action** is anything your robot does besides driving —

toggling the intake, shooting a ball, extending an arm.

Here are some actions that map well to a single button press:

  • A — turn the intake on or off
  • B — spin up the ball shooter
  • X — fire a ball into the goal
  • Y — raise or lower an arm

Each of these is something a driver triggers at the right moment during a match. One button, one job.

Why Separate Files?

You might expect all your code to live inside MecanumDrive.java. But in our project, each button gets its own file:

  • ButtonA_Action.java
  • ButtonB_Action.java
  • ButtonX_Action.java
  • ButtonY_Action.java

Each file has an execute() method that runs when its button is pressed. Why split things up?

Reuse. The same action file works in both TeleOp and Autonomous. When you record an auto path and press A at a certain spot, the robot calls the exact same ButtonA_Action.execute(). You write the code once and it works everywhere.

Teamwork. On a real FTC team, one student can work on the intake while another works on the shooter. Separate files mean separate work — no one accidentally overwrites someone else's code.

Clarity. MecanumDrive handles driving. ButtonA_Action handles the intake. Each file has one job. When something breaks, you know exactly where to look.

💡 Key Idea
Button action files keep mechanism code **separate** from drive

code. The same action works in both TeleOp and Autonomous.

For this lesson we will work in two files: ButtonA_Action.java and MecanumDrive.java.

State Machines

Our intake has two states: ON and OFF. Pressing the A button switches between them. It does not hold-to-run — one press turns it on, another press turns it off. This is called a toggle.

That toggle is a state machine — a program that:

  1. Tracks which state it is currently in
  2. Changes state based on inputs (like a button press)
💡 Key Idea
A **state machine** is a program that tracks which state it is

in and follows rules to switch between states.

Ours has two states and one input: 

State: OFF  →  press A  →  State: ON


State: ON   →  press A  →  State: OFF

How do we know which state we are in? We check the intake motor's current power. If it is spinning (power > 0), the intake is ON. If it is stopped (power = 0), the intake is OFF. The motor itself stores the state for us.

State machines show up everywhere in FTC. A ball shooter might have IDLE, SPINNING_UP, READY, and FIRING states. An arm might have STOWED, EXTENDING, and EXTENDED. Once you understand the pattern, you will see it in every mechanism.

Rising Edge Detection

Remember that while (opModeIsActive()) loop in your code? It runs about 50 times per second. Every time through, the program checks the gamepad, does math, and sends commands to the motors.

Here is the problem: if you check gamepad1.a directly, it will be true for every loop while the button is held down. Hold A for half a second and the toggle fires 25 times — flickering between ON and OFF rapidly.

The fix is called rising edge detection. You remember what the button was last loop. You only act when the button goes from false (released) to true (pressed) — the moment you first push it down. If the button was already true last loop, you skip it. 

loop 1:  A = false,  prevA = false  →  no change


loop 2:  A = true,   prevA = false  →  TRIGGER! (rising edge)
loop 3:  A = true,   prevA = true   →  skip (still held)
loop 4:  A = false,  prevA = true   →  no change (released)

In code, the check is: if (gamepad1.a && !prevA). That means "A is pressed NOW and was NOT pressed last time." After the check, you save the current state: prevA = gamepad1.a.

💡 Key Idea
**Rising edge detection** fires an action once when a button is

first pressed, not every loop while it is held down.

The Intake

Our robot has an intake — a spinning roller that pulls game balls into the robot. The intake can hold up to three balls in its hopper. Later lessons will cover how to shoot them, but first we need to turn the intake on and off.

The intake motor is called "intake_motor" in the hardware map. Setting its power to 1.0 spins it forward (pulling balls in). Setting it to 0.0 stops it.

Part 1 — Code the Toggle

✅ To Do
Replace the `Button Actions: Intake` comment block inside

execute() with the code from the green block:

>> Type This Code
// Get the intake motor from the hardware map


DcMotor intakeMotor = hardwareMap.dcMotor.get("intake_motor");




// Check if the motor is currently running
boolean isRunning = Math.abs(intakeMotor.getPower()) > 0.01;




// Toggle: if running, stop it; if stopped, start it
if (isRunning) {
    intakeMotor.setPower(0.0);
} else {
    intakeMotor.setPower(1.0);
}

Let's walk through what this does:

  1. Get the intake motor from the hardware map.
  2. Check its current power — if it is above 0.01, it is running.
  3. If running, stop it. If stopped, start it.

The Math.abs() call handles the case where power might be a tiny negative number due to floating-point math. Anything close to zero counts as stopped.

Notice we are NOT adding telemetry here. ButtonA_Action only runs once per button press, but the main loop calls telemetry.update() every frame — which would immediately overwrite anything we display here. We will add intake telemetry in the main loop instead.

Part 2 — Wire Up the Button

The action code is ready, but MecanumDrive does not call it yet. We need to add two things:

  1. A prevA variable to track the previous button state
  2. A rising-edge check that calls ButtonA_Action.execute()
✅ To Do
Click the link below to open MecanumDrive.java and jump to the

variable section:

[ MecanumDrive.java › Button Actions : Variables ]
✅ To Do
Type the line from the green block right below the

Add your button action variables below comment:

>> Type This Code
// Track previous A button state for rising edge detection


boolean prevA = false;

This creates a boolean that remembers whether A was pressed on the previous loop. It starts as false because no buttons are pressed when the program begins.

✅ To Do
Now click the link below to jump to the loop section. Type the

code from the green block right below the `Add your button action code below` comment:

[ Button Actions : Loop ]
>> Type This Code
// Rising edge: only trigger on the first frame A is pressed


if (gamepad1.a && !prevA) {
    new ButtonA_Action(this).execute();
}
prevA = gamepad1.a;  // remember this frame for next loop




// Show intake status on the Driver Hub every frame
DcMotor intakeMotor = hardwareMap.dcMotor.get("intake_motor");
telemetry.addData("Intake", Math.abs(intakeMotor.getPower()) > 0.01 ? "ON" : "OFF");

The first four lines handle the rising-edge toggle:

  • gamepad1.a && !prevA — A is pressed now AND was not pressed

last loop. This is the rising edge.

  • new ButtonA_Action(this).execute() — create a ButtonA_Action

and run its execute() method.

  • prevA = gamepad1.a — save the current button state for the

next loop.

The last two lines add telemetry. Because this runs every loop iteration (not just when A is pressed), the intake status stays visible on the Driver Hub at all times.

The this keyword passes your entire OpMode to ButtonA_Action. That is how it gets access to hardwareMap and telemetry without you wiring them up manually.
