Auto vs TeleOp

Two Halves of Every Match

An FTC match has two phases that run back-to-back:

  1. Autonomous (Auto) — 30 seconds. No human input. Your

code drives the robot all by itself.

  1. TeleOp — 2 minutes. The driver takes over with the

gamepad and controls the robot in real time.

Your MecanumDrive.java file is a TeleOp OpMode — it reads the gamepad every loop iteration and sends motor commands. But during the Autonomous phase the gamepad is ignored. The robot needs a completely separate program that says "drive forward, turn left, pick up that game piece" without any human help.

💡 Key Idea
**TeleOp** = human drives with a gamepad.

Autonomous = code drives itself — no gamepad, no driver. Every match starts with 30 seconds of Auto, then 2 minutes of TeleOp.

How Java Tells the Difference

FTC uses annotations — special labels above a class — to mark each OpMode: 

@TeleOp(name = "MecanumDrive")     // shows in TeleOp menu


public class MecanumDrive extends LinearOpMode { ... }




@Autonomous(name = "SimpleAuto")    // shows in Autonomous menu
public class SimpleAuto extends LinearOpMode { ... }

When the driver selects an OpMode on the Driver Hub, the FTC system checks which annotation you used and puts it on the right list. TeleOp OpModes appear under the TeleOp tab; Autonomous OpModes appear under the Autonomous tab.

💡 Key Idea
`@TeleOp` and `@Autonomous` are **annotations** — labels that

tell the FTC system which menu an OpMode belongs to.

What Makes an Auto Different?

Both TeleOp and Auto OpModes are Java classes that extend LinearOpMode. They both have a runOpMode() method. The difference is what happens inside that method.

In TeleOp, your while (opModeIsActive()) loop runs continuously, reading the gamepad each time: 

while (opModeIsActive()) {


    double drive = -gamepad1.left_stick_y;
    // ... send powers to motors
}

In Auto, there is no loop reading the gamepad. Instead you write a sequence of commands: 

waitForStart();


intakeMotor.setPower(1.0);   // turn on intake
sleep(3000);                  // wait 3 seconds
intakeMotor.setPower(0.0);   // turn off intake

The sleep() method pauses your program for a given number of milliseconds. During that pause, the motors keep doing whatever you last told them to do. This is the simplest way to write an auto — run a motor, wait, stop it.

💡 Key Idea
Auto OpModes use **sequences** — do this, wait, do that.

TeleOp OpModes use a loop — read gamepad, act, repeat.

💜 Hint
`sleep(3000)` = pause for 3000 milliseconds = 3 seconds.

The robot is NOT frozen during a sleep — motors keep spinning at whatever power you last set.

Part 2 — Write the Auto Sequence

Right now runOpMode() is empty. You will write a simple autonomous routine: turn on the intake for 3 seconds, then stop it.

✅ To Do
Replace the `Auto vs TeleOp` comment block inside

runOpMode() with the code from the green block:

>> Type This Code
// Get the intake motor


DcMotor intakeMotor = hardwareMap.dcMotor.get("intake_motor");




// Send a ready message to the Driver Hub
telemetry.addData("Status", "SimpleAuto ready");
telemetry.update();




// Wait for the driver to press START
waitForStart();




// === Autonomous sequence ===
// Step 1: Turn on the intake
intakeMotor.setPower(1.0);
telemetry.addData("Step", "Intake ON");
telemetry.update();




// Step 2: Wait 3 seconds while the intake runs
sleep(3000);




// Step 3: Stop the intake
intakeMotor.setPower(0.0);
telemetry.addData("Step", "Intake OFF — Done!");
telemetry.update();

Let's break down what this does:

  1. Get the motor — just like TeleOp, you pull the intake

motor from the hardware map by name.

  1. telemetry — display a "ready" message before the match

starts so drivers know the robot is initialized.

  1. waitForStart() — your code pauses here until the driver

presses the START button. This is required in both TeleOp and Auto.

  1. setPower(1.0) — spin the intake at full power.
  2. sleep(3000) — wait 3000 ms (3 seconds). The intake keeps

spinning during this pause.

  1. setPower(0.0) — stop the intake. Auto is done!

Part 3 — Run Your Auto

Switch to the field (press Escape to leave the PIT). On the Driver Hub, select SimpleAuto from the Autonomous menu and press START.

✅ To Do
Watch the robot. The intake should spin for exactly 3 seconds,

then stop. Check the telemetry messages on the Driver Hub.

If the intake did not spin, double-check:

  • Did you use the correct motor name "intake_motor"?
  • Did you remember waitForStart()?
  • Is the power set to 1.0 (not 0.0)?

Why This Matters

This "sleep-based" auto is the simplest possible approach. It has a big weakness: the robot has no idea where it is. If it bumps into something and gets knocked off course, it will not correct itself.

Later lessons will teach you path-following with sensors (OTOS, AprilTags) and the Pedro Pathing library — where the robot knows its position and follows a precise path. But every autonomous program starts with the same building blocks you just learned:

  • @Autonomous annotation
  • waitForStart()
  • A sequence of motor commands and sleep() calls
💡 Key Idea
Sleep-based auto is simple but fragile — the robot cannot

correct for bumps or drift. Path-following (later lessons) uses sensors to stay on course.
