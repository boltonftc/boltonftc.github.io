Telemetry

What is Telemetry (and Why It Matters)?

Telemetry is how your robot talks back to you while it’s running. It sends information from the robot controller (the code and hardware) to the Driver Station so you can see what’s happening in real time.

💡 Key Idea
**Telemetry** is the system that sends data from your robot

back to the Driver Station screen.

In FTC, your robot is constantly making decisions—reading sensors, checking button inputs, and controlling motors. But without telemetry, all of that is invisible. The robot would feel like a “black box.”

Telemetry turns that black box into something you can see and understand.

Why Use Telemetry?

💡 Key Idea
**Telemetry** is your main debugging tool when working on a robot.

Here are the most common reasons teams use telemetry:

  • Debugging code

Check if your program is doing what you expect. Example: “Is the button press actually being detected?”

  • Reading sensor values

See live data from sensors like distance, color, or IMU. Example: “What distance is the sensor actually reading?”

  • Understanding robot behavior

Figure out why the robot is acting a certain way. Example: “Why isn’t the motor turning?”

  • Driver feedback during matches

Show useful info to drivers. Example: “Is the intake running?” or “What mode are we in?”

How Does Telemetry Work In FTC Code?

💡 Key Idea
Telemetry works by adding messages to a list, then sending them

to the Driver Station using telemetry.update().

Every loop of your program, you can:

  • Add lines of text
  • Add labeled data (like variables or sensor values)
  • Send everything to the screen
💜 Hint
If you don’t call telemetry.update(), nothing will appear!

Part 2 -- Variables, Booleans, and Button States

Before we go further, let's talk about variables. A variable is a named container that holds a piece of information. Think of it like a labeled box -- the label is the variable's name, and whatever is inside the box is its value. Your code can look inside the box at any time to see what is there, and the value can change as the program runs.

For example, gamepad1.a is a variable. Its name tells you it represents the A button on gamepad 1. Its value tells you whether that button is currently pressed or not.

💡 Key Idea
A **variable** is a named container that holds a value. The

name never changes, but the value inside can change while your program runs.

Gamepad buttons are one of the simplest types of input in your robot code. Each button is either pressed or not pressed, and your program needs a way to represent that.

In Java, this type is called boolean.

💡 Key Idea
A boolean is a value that can only be true or false.

Gamepad buttons use booleans. Each button is always in one of two states:

  • TRUE means the button is pressed
  • FALSE means the button is not pressed

For example:

  • When you press the A button, gamepad1.a becomes TRUE
  • When you release it, gamepad1.a becomes FALSE

There are no in-between values. It is always one or the other.
