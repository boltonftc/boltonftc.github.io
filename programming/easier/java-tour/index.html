The Package Declaration

[ package org.firstinspires.ftc.teamcode ]

Just below the gray comment block at the top of the file is the very first line of real code: 

package org.firstinspires.ftc.teamcode;

A package is Java's way of grouping related files together. Think of it like a folder name on your computer — it tells the robot's Controller Hub which team's code to run. All the files in your code package share the same package name: org.firstinspires.ftc.teamcode.

The dots are like slashes in a folder path: org / firstinspires / ftc / teamcode. You will never need to change this line.

💡 Key Idea
A **package** is a named group of Java files. All your robot

code lives in the package org.firstinspires.ftc.teamcode. Think of it as the folder address for your team's code.

The Imports

[ FTC SDK Imports ]

Directly below the package declaration are a series of import lines, organized into labeled sections: 

import com.qualcomm.robotcore.eventloop.opmode.LinearOpMode;


import com.qualcomm.robotcore.eventloop.opmode.TeleOp;
import com.qualcomm.robotcore.hardware.DcMotor;

Java cannot use a class unless you tell it where to find it. import is how you do that — each line brings in one class from the FTC SDK (the official robotics library). As you add new features in later lessons, you will see new imports appear at the top of the file.

The @TeleOp Annotation

[ @TeleOp(name ]

Just above the class declaration there is a line with an @ symbol: 

@TeleOp(name = "MecanumDrive", group = "Starter")

The @ marks a Java annotation — a label that attaches extra information to code without being a line the robot actually runs. The @TeleOp annotation tells the FTC Driver Hub that this program is a driver-controlled OpMode and makes its name show up in the program list on the tablet.

name = "MecanumDrive" is what the driver sees and selects before a match. If you change that string, the name on the Driver Hub changes.

The Class Declaration

[ public class MecanumDrive ]

The next line opens the class: 

public class MecanumDrive extends LinearOpMode {

Here is what each piece means:

  • public — other parts of the program can see this class
  • class MecanumDrive — the name of the class
  • extends LinearOpMode — your class inherits all the

built-in FTC robot tools already written for you: reading hardware, accessing the gamepad, checking match state, etc.

  • { — opens the class body; everything until the last } in

the file belongs to this class

You will sometimes hear extends described as an "is-a" relationship. MecanumDrive is a LinearOpMode — it has all the same built-in capabilities, plus whatever you add on top.

We will go deeper on classes in a later lesson. For now, just know: the class is the container for everything.

💡 Key Idea
A **class** in Java is a named container that holds related

code together. MecanumDrive extends LinearOpMode means your program gets all the built-in FTC robot tools for free — you just add your own robot logic on top.

The runOpMode() Method

[ public void runOpMode ]

Inside the class, you will see this: 

@Override


public void runOpMode() throws InterruptedException {

This is the method that contains everything your robot does. A method is a named block of instructions. When the FTC framework needs your robot to run, it calls runOpMode() and executes every line inside its { and }.

A few things on these lines you do not need to dig into yet:

@Override — You are replacing (overriding) a built-in version of runOpMode() that lives in LinearOpMode. Your version wins. The @Override annotation is just a safety check that says "I know I am replacing something."

throws InterruptedException — Java requires this declaration on any method that uses sleep(). The computer needs to see it. Ignore what it means for now.

💡 Key Idea
A **method** is a named block of instructions. `runOpMode()`

is the main method for FTC — everything your robot does during a match is written inside it. The FTC framework calls it automatically when the driver presses START.

Connecting to Hardware

[ hardwareMap.dcMotor.get ]

The first lines inside runOpMode() grab the four drive motors: 

DcMotor backLeft  = hardwareMap.dcMotor.get("back_left_motor");


DcMotor frontLeft = hardwareMap.dcMotor.get("front_left_motor");

The hardwareMap is a built-in object that contains every piece of hardware plugged into the Control Hub. .dcMotor.get looks up a motor by the name label configured on the Driver Hub. Those names — back_left_motor, front_left_motor, and so on — must exactly match the labels on the physical wiring of the real robot.

You got a look at the physical robot in an earlier lesson. The motors have name stickers on them. If the string in the code does not match the sticker, the program will crash. Getting these names right is one of the most common setup bugs on a real robot.

Naming Conventions — camelCase and PascalCase

While you are looking at the hardware setup, notice the pattern of how things are named: 

DcMotor backLeft  = hardwareMap.dcMotor.get("back_left_motor");

backLeft starts with a lowercase letter and each new word after the first is capitalized: backLeft, frontLeft, backRight. This convention is called camelCase because the capital letters in the middle look like camel humps.

In Java, camelCase is used for variables and methods — the names of things that hold values or do work: backLeft, frontLeft, opModeIsActive, setDirection.

DcMotor starts with a capital letter with each new word also capitalized: DcMotor, LinearOpMode, MecanumDrive. This is called PascalCase (every word capitalized, no separator).

In Java, PascalCase is used for class names — the names of types and blueprints: DcMotor, LinearOpMode, MecanumDrive.

You will see these patterns throughout the codebase. When you write your own variables, use camelCase. When you name a class, use PascalCase.

💡 Key Idea
**camelCase** — first word lowercase, each following word

capitalized. Used for variables and methods: backLeft, setDirection, opModeIsActive.

PascalCase — every word capitalized. Used for class names: DcMotor, LinearOpMode, MecanumDrive.

Reversing the Left Motors

[ backLeft.setDirection ]

Right below the hardware setup: 

backLeft.setDirection(DcMotor.Direction.REVERSE);


frontLeft.setDirection(DcMotor.Direction.REVERSE);

The left-side motors are mounted as a mirror image of the right-side motors. If you tell both sides to spin "forward" without reversing the left side, the left wheels push the robot backward while the right wheels push it forward. The call to setDirection(REVERSE) fixes this so all four wheels agree on which way is "forward."

The Start Line — waitForStart()

[ waitForStart ]

After all the setup, there is one critical line: 

waitForStart();

This is the dividing line between init and loop — the same boundary you saw in the Code Orientation lesson. Everything above waitForStart() runs once when the program loads. The code pauses here and waits for the driver to press START on the Driver Hub.

The Operational Loop

[ while (opModeIsActive ]

Immediately below waitForStart(): 

while (opModeIsActive()) {

This is the loop — a block of code that repeats over and over until the match ends. opModeIsActive() is a built-in method that returns true while the match is running and false when it ends or the driver presses STOP. The while keeps checking: *"Is the match still going? If yes, run the code inside the braces again."*
