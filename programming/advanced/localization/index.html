Localization

The Best of Both Worlds

You now have two sensors that can tell the robot where it is:

  • OTOS — tracks position continuously by watching the

ground. Always available, updates 1000 times per second. But it drifts — small errors pile up and the reported position slowly wanders away from reality.

  • Limelight + AprilTags — calculates position by seeing

field landmarks. Zero drift — every detection is perfectly fresh. But it only works when tags are visible, and it only updates about 11 times per second.

Each sensor is strong where the other is weak. What if you could combine them?

That is localization — the art of knowing exactly where you are, all the time, by fusing multiple sources of information into one reliable answer.

💡 Key Idea
**Localization** combines multiple sensors to get a better

position estimate than any single sensor alone. The OTOS provides continuous tracking, and AprilTags provide drift-free corrections whenever visible.

What Is Localization?

The word localization comes from "locale" — to figure out your location. In robotics, it answers the most fundamental question: "Where am I right now?"

This sounds simple, but it is surprisingly hard. No single sensor gives a perfect answer:

| Sensor | Strength | Weakness | | --- | --- | --- | | Motor Encoders | Cheap, always available | Wheel slip ruins everything | | IMU (Gyroscope) | Great for rotation | No position info, drifts over time | | OTOS | X, Y, heading — all in one | Accumulates drift | | AprilTags | Zero drift, absolute position | Only works when tags are visible |

The solution used by virtually every competitive FTC team is sensor fusion — combining the best qualities of each sensor into a single, continuously updated position estimate.

The Fusion Strategy

The strategy we will implement is simple but effective:

  1. Use the OTOS as the primary tracker. It runs at high

speed and always has a position to report. Between AprilTag sightings, the OTOS is your only source of truth.

  1. When the Limelight sees an AprilTag, snap the position back to reality.

The MegaTag2 position is more accurate than the drifted OTOS, so we reset the OTOS's X and Y to match what Limelight reports.

  1. Keep the OTOS heading. The OTOS has a built-in IMU that

tracks rotation very accurately. The Limelight's heading estimate is noisier. So we trust the OTOS for heading and only correct X and Y from AprilTags.

Think of it like navigating a hike with a compass and a map. The compass (OTOS) tells you your direction continuously, and the map landmarks (AprilTags) tell you your exact position whenever you spot one. Between landmarks, you dead-reckon with the compass. At each landmark, you update your position. The result is much more accurate than either tool alone.

💡 Key Idea
**Fusion strategy:** OTOS tracks continuously (with drift).

When AprilTags are visible, reset OTOS X/Y position to the Limelight's reading. Keep OTOS heading (more stable than vision heading). This eliminates drift while maintaining continuous tracking.

Why Not Just Use AprilTags for Everything?

Good question. If AprilTags give perfect position, why bother with the OTOS at all?

Because AprilTags are intermittent. During a 30-second autonomous routine, the robot might only see tags for a few seconds total — when it is facing a wall at the right distance. The rest of the time, it is driving across the field with no tags in view. During those blind periods, the OTOS is the only thing keeping track of where the robot is.

And 11 frames per second is too slow for precise driving control. The OTOS updates at 1000 Hz — nearly 100 times faster. Your motion code needs that high-speed feedback to make smooth, accurate movements. The AprilTag corrections are like periodic GPS fixes — they keep the OTOS from wandering too far off course.

When to Trust (and Distrust) a Correction

Not every AprilTag detection is worth trusting. There are situations where the Limelight's position estimate might be wrong:

Distance Matters

At long range (beyond 6 feet), the tag is small in the image and the position estimate becomes noisy. A good rule: only accept corrections when the average tag distance is under 72 inches (6 feet).

Tag Count Matters

Seeing two or more tags is much more reliable than one. With two tags, MegaTag2 can cross-check the position from multiple angles. With one tag, there is only one measurement to work with.

Heading Quality Matters

Remember, MegaTag2 uses your IMU heading. If the heading has drifted badly (which is rare in 30 seconds but possible after many minutes), the MegaTag2 position will be wrong too. This is another reason to keep autonomous routines short and efficient.

💡 Key Idea
**Only trust AprilTag corrections when:** the tags are within

range (under 72 inches), the heading is fresh, and ideally more than one tag is visible. Filter out unreliable detections to avoid making the position estimate worse.

You already have OTOS and Limelight telemetry from the previous two lessons. Now you will add the fusion logic that ties them together — the OTOS tracks continuously, and AprilTag detections snap the position back to reality.

The key change is simple: when MegaTag2 gives us a valid position, we call otos.setPosition() to reset the OTOS's X and Y. The OTOS will continue tracking from that corrected position, and the drift clock starts over from zero.

>> Type This Code
// === Fused Localization ===


// Feed the OTOS heading to Limelight for MegaTag2
SparkFunOTOS.Pose2D currentPos = otos.getPosition();
limelight.updateRobotOrientation(currentPos.h);




// Check for AprilTag correction
LLResult llResult = limelight.getLatestResult();
boolean corrected = false;




if (llResult != null && llResult.isValid()) {
    LLResultTypes.BotPose mt2 = llResult.getBotpose_MT2();




    // Only trust corrections within 72 inches
    if (mt2 != null && mt2.getTagCount() > 0
            && mt2.getAvgDist() <= 72.0) {
        // Reset OTOS X/Y to Limelight position
        // Keep OTOS heading (more stable than vision)
        otos.setPosition(new SparkFunOTOS.Pose2D(
                mt2.getX(), mt2.getY(), currentPos.h));
        corrected = true;
    }
}




// Re-read position (may have been corrected)
SparkFunOTOS.Pose2D fused = otos.getPosition();




// Display fused position
telemetry.addData("Pos X", "%.1f in", fused.x);
telemetry.addData("Pos Y", "%.1f in", fused.y);
telemetry.addData("Heading", "%.1f°", fused.h);
telemetry.addData("Source", corrected ? "OTOS + AprilTag" : "OTOS only");




// Show tag info when available
if (llResult != null && llResult.isValid()) {
    telemetry.addData("Tags", "%d", llResult.getFiducialResults().size());
    for (LLResultTypes.FiducialResult tag : llResult.getFiducialResults()) {
        telemetry.addData("  Tag", "#%d  %.1f°", tag.getFiducialId(),
                tag.getTargetXDegrees());
    }
} else {
    telemetry.addData("Tags", "none");
}

Let's walk through the fusion logic step by step:

  1. Read OTOS — Get the current (possibly drifted) position

from the OTOS.

  1. Feed heading — Send the OTOS heading to the Limelight

so MegaTag2 can use it.

  1. Check for correction — If the Limelight sees tags within

72 inches, it has a trustworthy MegaTag2 position.

  1. Apply correction — Call otos.setPosition() with the

Limelight's X and Y but the OTOS's heading. This "snaps" the OTOS back to the correct position.

  1. Re-read — Read the OTOS again (now corrected) and

display the fused result.

  1. Show source — The telemetry shows whether the current

reading was corrected by AprilTags or is OTOS-only. This helps you see the fusion in action.

How the Pros Do It

The localization code you just wrote follows the same architecture used by top-performing FTC teams:

  1. OTOS at the core — continuous, high-speed position

tracking as the primary source

  1. Limelight MegaTag2 corrections — when tags are visible

and within range, snap the position to the camera-derived estimate

  1. OTOS heading trusted — the built-in IMU provides stable

heading that MegaTag2 relies on

This is the same strategy implemented in the competition-ready FusedOTOSLocalizer class that powers autonomous path following. The telemetry version you built today is simpler (it does not integrate with a path planner), but the core fusion logic is identical.

In future lessons, you will learn about pose estimation (where the robot plans to go, not just where it is) and path following (making the robot drive precise autonomous routes). Those systems will plug directly into this localization foundation.
