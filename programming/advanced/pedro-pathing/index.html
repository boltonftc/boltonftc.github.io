Pedro Pathing: Capture Mode

From Sleep-Based Auto to Path Following

In the Auto vs TeleOp lesson, you wrote a simple autonomous routine that used sleep() to time motor commands. That works, but it is blind — the robot has no idea where it is. If it bumps a wall or another robot, it drifts off course with no way to recover.

Pedro Pathing is a path-following library that solves this problem. It uses your robot's localizer (the OTOS sensor and AprilTag data you set up in earlier lessons) to know exactly where the robot is at all times. You define a path — a series of waypoints — and Pedro drives the robot along that path, correcting for drift as it goes.

💡 Key Idea
**Pedro Pathing** = real-time path following with drift correction.

You define where the robot should go; Pedro drives it there using sensor feedback from the localizer.

The Capture Panel (Teach Pendant)

Writing Pedro Pathing code by hand means typing exact coordinates and headings for every waypoint. That is tedious and error-prone. Instead, the simulator has a Capture Panel — a teach-pendant-style tool that lets you drive the robot to each position and record it with a button press.

Here is how it works:

  1. You drive the robot around the field with the gamepad.
  2. At each important position, you press LB (Left Bumper) to

record the robot's pose (X, Y, heading).

  1. At positions where the robot should perform an action, you

press A, B, X, or Y to record that action.

  1. When you are done, you press Export and the simulator

writes a complete Java file with all the Pedro Pathing code.

No typing coordinates. No guessing headings. You drive the path you want, and the tool generates the code.

💡 Key Idea
The **Capture Panel** is a teach pendant: you drive the robot,

record poses with LB, record actions with A/B/X/Y, and export a ready-to-run Java autonomous file.

Part 1 — Enter Capture Mode

Capture mode is available from the field (not the PIT).

✅ To Do
1. Press Escape to go to the field if you are in the PIT.
  1. On the Driver Hub panel, find and press the Capture button.
  2. The Capture Panel will appear on screen with a list of

recorded poses (currently empty) and control buttons.

The panel shows:

  • Pose list — each recorded waypoint appears here
  • LB / A / B / X / Y indicators — show what each button

records

  • Clear — erase all recorded poses and start over
  • Export — generate the Java file

Part 2 — Record a Path

Now drive a path around the field. A good first path might be:

  1. Drive forward about 3 feet
  2. Turn 90 degrees left
  3. Drive forward to a game element
  4. Fire the intake (Button A action)
  5. Drive to the scoring zone
✅ To Do
1. Use the left stick to drive the robot to your first waypoint.
  1. Press LB to record the pose. You will see it appear in

the Capture Panel's pose list.

  1. Drive to the next position and press LB again.
  2. When you reach a spot where the robot should run an action,

press the corresponding button: - A = intake toggle (ButtonA_Action) - B = flywheel toggle (ButtonB_Action) - X = indexer fire (ButtonX_Action) - Y = ButtonY_Action

  1. Record at least 4-5 poses to make an interesting path.
💜 Hint
You do not need dozens of waypoints. Pedro Pathing draws smooth

lines between poses. 4-6 well-placed poses make a clean path. More poses = more precise control, but also more complexity.

The Capture Panel shows each entry as either a Pose (position waypoint) or an Action (button press at that location). You can see the coordinates update as you add entries.

Part 3 — Export the Auto File

Now export your recorded path as a Java autonomous file.

✅ To Do
1. In the Capture Panel, find the **class name** text field.

Type a name for your auto — for example MyFirstAuto.

  1. Press the Export button.
  2. The simulator writes a new .java file into your code package

with the Pedro Pathing path you recorded.

The exported file is a complete, compilable Java class with:

  • @Autonomous annotation
  • Pedro Pathing Follower setup with tuned PID constants
  • Pose objects for every waypoint you recorded
  • PathChain connecting all the poses with Bezier curves
  • Button actions inserted at the right points
  • A while (opModeIsActive() && follower.isBusy()) loop that

follows the path

You did not have to write any of that — the Capture Panel generated it all from your drive session.

💡 Key Idea
The **Export** button generates a full Pedro Pathing Java file.

All the path math, PID setup, and action triggers are created automatically from the poses you recorded.

Part 4 — Run Your Auto

Test the exported autonomous path.

✅ To Do
1. Press Escape to close the Capture Panel if it is still open.
  1. On the Driver Hub, switch to the Autonomous menu.
  2. Select the auto file you just exported (the class name you

typed).

  1. Press START and watch the robot follow your recorded path.

The robot should follow the same route you drove, hitting each waypoint and performing each action at the right spot. Watch the telemetry — it shows the current path phase and the robot's real-time position.

If the path looks wrong:

  • Go back into Capture mode and re-record
  • Make sure you pressed LB at each key position
  • Try fewer poses for simpler paths

Reading the Generated Code

Open the exported Java file in the code editor to see what Pedro Pathing code looks like. Here are the key pieces:

Poses

Pose pose1 = new Pose(-0.1, -3.5, Math.toRadians(91.4));


Pose pose2 = new Pose(-32.9, -3.5, Math.toRadians(91.4));

Each Pose has three values: X (inches from field center), Y (inches from field center), and heading (direction the robot faces, in radians). These were captured from the exact positions you drove to.

Path Chains

PathChain chain1 = follower.pathBuilder()


    .addPath(new BezierLine(pose1, pose2))
    .setLinearHeadingInterpolation(
        Math.toRadians(91.4), Math.toRadians(91.4))
    .build();

A PathChain connects poses with smooth Bezier curves. The heading interpolation controls how the robot rotates between waypoints — setConstantHeadingInterpolation keeps the same heading, setLinearHeadingInterpolation smoothly rotates between two headings.

The Follower Loop

follower.followPath(chain1);


while (opModeIsActive() && follower.isBusy()) {
    follower.update();
    telemetry.addData("Phase", "Chain 1");
    telemetry.update();
    idle();
}

follower.followPath() starts path execution. The while loop calls follower.update() every cycle, which reads the localizer, calculates motor powers, and drives toward the next waypoint. When the robot reaches the end, isBusy() returns false and the loop exits.

💡 Key Idea
Pedro Pathing code has three main parts: **Poses** (where to go),

PathChains (how to connect them), and the Follower loop (drive and correct until you arrive).

What Happens on a Real Robot?

Everything you just did works exactly the same on a real FTC robot with two small differences:

  1. Localizer hardware — instead of a simulated OTOS sensor,

you wire a real SparkFun OTOS to the Control Hub.

  1. PID tuning — the PID constants in the generated file may

need adjustment for your real robot's weight and wheel grip.

The Pedro Pathing library, the Follower class, the PathChain system — all of that is the same code running on the real robot's Control Hub. That is why teams prototype autonomous paths in the simulator first: it is much faster to iterate than driving a physical robot back and forth across a field.
