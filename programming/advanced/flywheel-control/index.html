Flywheel Control: Feedforward + PID

The Problem

You built a flywheel shooter. You set its power to 1.0 and it spins up to 312 RPM. You fire a ball — the RPM drops 40%. You wait... and wait... and eventually it recovers. Then you fire again.

What if you want to fire three balls quickly, one after another? With a simple setPower(1.0) the first shot is strong, the second is weak, and the third barely makes it. Why? Because the flywheel has not recovered between shots.

💡 Key Idea
**The core problem:** `setPower(1.0)` is **open-loop** control.

You set a fixed power and hope for the best. The motor has no idea it slowed down after a shot — it just keeps applying the same voltage. Recovery is slow because the motor is already at "full power" and has nothing extra to give.

The Smart Way: Tell the Motor What to Do

Imagine you are driving on cruise control set to 60 mph. The car does not just hold the gas pedal in one spot — it measures the speed with the speedometer and adjusts the throttle. Going uphill? More gas. Going downhill? Less gas.

But what if the car already knew roughly how much gas it needs for 60 mph on flat road? It could start there and only make small adjustments. That is much faster than starting from zero and guessing.

That is the idea behind feedforward + feedback control:

  1. Feedforward — "I already know roughly what power to set

for this target speed" (the starting point)

  1. Feedback (P) — "Let me fine-tune based on what the

encoder actually reads" (the correction)

💡 Key Idea
**Feedforward** gives you a smart starting point based on

physics. Feedback (the P term) corrects for real-world disturbances like shots, friction, and battery drain. Together they are faster and more stable than either one alone.

The Formula

The total motor power is: 

power = kS + kV × targetRPM + kP × error

| Constant | Name | What It Does | |----------|------|-------------| | kS | Static friction | Minimum power to overcome friction and start spinning | | kV | Velocity constant | Maps target RPM directly to power (the motor curve) | | kP | Proportional gain | Corrects based on the current error (target − actual) |

You will tune each one individually, in order. This is the same approach used by FTCLib, Road Runner, and most competitive FTC teams.

Part 1 — Finding kS: "How Much Power Just to Start?"

Every motor has internal friction. If you set power to 0.01, the shaft does not move. You need a minimum power just to overcome friction and get the flywheel barely turning. That minimum power is kS.

Think of it like pushing a heavy box on carpet. You push lightly — nothing happens. You push a bit harder — still nothing. Then at some threshold the box starts to slide. That threshold is kS.

💡 Key Idea
**kS (static friction):** The minimum motor power needed to

overcome friction and start the shaft turning. Below kS, the motor buzzes but does not spin.

Part 2 — Finding kV: "How Much Power Per RPM?"

Now you know the friction floor (kS). The next question: once the motor is spinning, how much additional power do you need for each RPM of target speed?

This is the velocity constant — it maps target RPM to the power needed to sustain that speed at steady state. For a DC motor, this relationship is nearly linear: 

power_for_speed = kS + kV × targetRPM

💡 Key Idea
**kV (velocity constant):** The slope of the motor's

power-to-speed curve. It tells you how much power each additional RPM costs. kV = (1.0 - kS) / maxRPM is a good starting estimate.

Calculate Your Starting kV

Your motor's max RPM is 312 at full power (1.0). So: 

kV = (1.0 - kS) / 312

If kS = 0.06: 

kV = (1.0 - 0.06) / 312 = 0.94 / 312 ≈ 0.00301

💡 Key Idea
**Feedforward is proactive.** It does not wait for an error —

That is where kP comes in.

Part 3 — Adding kP: "Correct What Feedforward Misses"

Feedforward gets you 95% of the way there. But it cannot react to disturbances — shots, battery voltage changes, friction changes. For that, you need a small amount of feedback.

The P (proportional) term measures the current error and adds extra power proportional to that error: 

error = targetRPM - actualRPM


correction = kP × error
total_power = kS + kV × targetRPM + kP × error

After a shot drops the RPM by 100, kP kicks in with extra power to bring it back faster. Once the error shrinks to near zero, kP fades away and feedforward holds steady.

💡 Key Idea
**kP (proportional gain):** Adds a correction proportional to

the current error. Big error → big boost. Small error → small boost. It handles disturbances that feedforward cannot predict.

Add kP to Your Code

First, you need a variable to hold the target RPM that persists between loop iterations (so the right stick adjustment is "sticky"). Add these in the **Flywheel Control : Variables** section (above the main loop):

Now replace your flywheel control loop code with the full feedforward + P controller:

[ MecanumDrive.java › Flywheel Control : Loop ]
>> Type This Code
// ── Flywheel Control ──────────────────────────────


// ┌─────────────────────────────────────────────┐
// │  FEEDFORWARD + P CONSTANTS                  │
// │                                             │
// │  kS: 0.06    (static friction — Step 1)     │
// │  kV: 0.00301 (velocity constant — Step 2)   │
// │  kP: 0.005   (proportional gain — tune now) │
// │                                             │
// │  Right stick Y: adjust target RPM (sticky)  │
// │  Up = faster, down = slower, center = hold  │
// └─────────────────────────────────────────────┘
if (flywheelOn) {  // flywheelOn is already set by the Flywheel Trim section above
    double kS = 0.06;
    double kV = 0.00301;
    double kP = 0.005;




    // Compute time since last frame for sticky stick adjustment
    long now = System.nanoTime();
    double dtSec = (now - lastControlTime) / 1_000_000_000.0;
    lastControlTime = now;




    // Right stick Y adjusts target RPM (sticky — holds value when released)
    double stickY = -gamepad1.right_stick_y;
    if (Math.abs(stickY) > 0.05) {
        targetRPM += stickY  60.0  dtSec;  // 60 RPM/sec adjust rate
        targetRPM = Math.max(100.0, Math.min(312.0, targetRPM));
    }




    // Feedforward: predict the power needed for this RPM
    double feedforward = kS + kV * targetRPM;




    // Feedback: correct based on current error
    // (rpm already computed in the Encoder RPM section above)
    double error = targetRPM - rpm;
    double correction = kP * error;




    // Total power = feedforward + feedback
    double power = feedforward + correction;




    // Clamp to valid motor range [0, 1]
    power = Math.max(0.0, Math.min(1.0, power));
    shooterMotor.setPower(power);




    telemetry.addData("Target RPM", "%.0f", targetRPM);
    telemetry.addData("Actual RPM", "%.1f", rpm);
    telemetry.addData("Error", "%.0f", error);
    telemetry.addData("FF Power", "%.3f", feedforward);
    telemetry.addData("P Correction", "%.3f", correction);
    telemetry.addData("Total Power", "%.3f", power);
}

  • If recovery after a shot is too slow → increase kP
  • If the green line oscillates (bounces up and down) →

  • Sweet spot: fast recovery with no visible bouncing
💡 Key Idea
**Why not just use a big kP?** Because feedforward already

provides the baseline power, kP only needs to handle small disturbances. This means kP can be small, which prevents oscillation. Compare this to pure P-only control where kP has to do all the work — it must be big enough to reach the target, but then it overshoots and oscillates.

What About PID?

You may have heard of PID control (Proportional, Integral, Derivative). That is a valid approach, but for flywheels the feedforward + P method is simpler and better:

| | Pure PID | Feedforward + P | |---|---|---| | Spin-up | Slow — P must discover correct power | Fast — feedforward knows immediately | | Steady-state gap | Needs I term to close it | Feedforward handles it directly | | Oscillation risk | High if P is too aggressive | Low — P only handles small corrections | | Tuning complexity | 3 interacting gains (Kp, Ki, Kd) | 3 independent values (kS, kV, kP) |

In PID, the I (integral) term slowly accumulates past error to close the steady-state gap — but feedforward eliminates that gap instantly. The D (derivative) term dampens oscillation — but since kP is small (feedforward does the heavy lifting), oscillation is rarely a problem.

Competitive FTC teams (and libraries like FTCLib and Road Runner) use feedforward + P as the standard approach for velocity control.

💜 Hint
**When IS full PID useful?** PID shines for **position control**

— holding an arm at a specific angle, or keeping the robot at a specific heading. In those cases there is no simple feedforward model, so you need I and D to handle steady-state error and oscillation. Different tool for a different job.

Tuning Summary

Here is the full tuning process, start to finish:

  1. Find kS: Slowly increase power until the flywheel barely

spins. That is your kS. (Typical: 0.05–0.12)

  1. Calculate kV: kV = (1.0 - kS) / 312. Test at your

target RPM, fine-tune until the steady-state error is small. (Typical: 0.0025–0.0035)

  1. Tune kP: Start at 0.005. Increase until shot recovery is

fast. Decrease if you see oscillation. (Typical: 0.002–0.015)

💜 Hint
**These values transfer to real robots.** The kS, kV, kP

tuning process is the same on a physical robot with the same motor. You may need to re-tune kS (real friction is different), but the process and code are identical.

What You Learned

  • Open-loop control (setPower) has no feedback — it

cannot react to disturbances like shots

  • Feedforward (kS + kV) predicts the right power from

physics — no error needed

  • kS overcomes static friction (the floor)
  • kV maps target RPM to power (the slope)
  • kP corrects for disturbances (the safety net)
  • Feedforward + P is simpler and faster than full PID for

velocity control

  • This is the standard approach used by FTCLib, Road Runner,

and competitive FTC teams

Where Else Is This Used?

Feedforward + feedback controllers are everywhere:

  • FTC drivetrains — feedforward for velocity, PID for

heading correction

  • Industrial motors — factory conveyor belts, CNC machines
  • Cruise control — throttle map (feedforward) + speed

sensor correction (P)

  • Drone rotors — RPM model (feedforward) + gyro

stabilization (PID)

The concepts you just learned — measuring, predicting, and correcting — apply to any system where you have a target, a sensor, and an actuator.
